# CourseForge Project Feature Guide

## 1. What This Project Is

CourseForge is a browser-based course authoring tool with a React frontend and a FastAPI backend. Authors build courses slide-by-slide, assemble those slides from reusable content and assessment blocks, preview the finished experience in-browser, and export the same authored structure to SCORM 1.2, SCORM 2004, and xAPI-compatible runtime packages.

At a practical level, the product has four major responsibilities:

1. Authoring: a visual editor for slides, backgrounds, blocks, and quiz settings.
2. Importing: convert PowerPoint, PDF, audio, and Storyline-derived content into editable CourseForge slides.
3. Runtime generation: transform authoring data into a normalized runtime course definition.
4. Packaging: produce ZIP exports containing HTML, runtime JS, serialized course data, manifests, and extracted media files.

## 2. Core Architecture

### Frontend

- Main editor entry point: `courseforge-frontend/src/App.jsx`
- Main styling: `courseforge-frontend/src/App.css`
- Block implementations: `courseforge-frontend/src/components/blocks/`
- Shared UI widgets: `courseforge-frontend/src/components/ui/`
- Browser persistence: `courseforge-frontend/src/utils/storage.js`

The frontend owns authoring state. Each slide contains an `elements` array, and each element is one block. Most edits are local state updates inside `App.jsx`, with block components calling `onUpdate(block.id, patch)` to mutate their own data.

### Backend

- API routes: `app/api/routes.py`
- SCORM/runtime transformer: `app/processors/scorm_builder.py`
- Import processors:
  - `app/processors/pptx_parser.py`
  - `app/processors/pdf_parser.py`
  - `app/processors/story_parser.py`

The backend does not own the authoring UI. Instead, it receives the current course JSON, converts it into a runtime course definition, then renders or packages the result.

## 3. Authoring Model

### Course

A course is defined by:

- `title`
- `slides`
- `policy`
  - `passingScore`
  - `maxAttempts`
  - `lockOnPass`
  - `lockOnExhaust`

In the editor, a draft also includes local browser metadata such as local course ID and save timestamps.

### Slide

Each slide typically contains:

- `id`
- `type: "slide"`
- `title`
- `elements`
- optional `background`
  - `{ type: "color", value: "#xxxxxx" }`
  - `{ type: "image", value: "data-uri-or-url" }`
- optional `bgAudio`

### Block

Each slide block is a typed object such as `text`, `image`, `quiz`, `tabs`, `canvas`, etc. `App.jsx` creates the default shape for each block in the `addBlock()` function, and the matching React component renders the editor for that shape.

## 4. Main User Flow

### Dashboard and Local Save

- `Dashboard.jsx` is the course picker / landing view.
- `storage.js` uses `localforage` to save authoring state in the browser.
- `createLocalCourse()` creates a new locally stored course.
- `saveCourseToBrowser()` updates an existing course and timestamps it.
- `getAllLocalCourses()` drives the dashboard list.
- `deleteLocalCourse()` removes a saved draft.

This means users can keep working without exporting immediately, and the editor behaves like a lightweight local project manager.

### Editing Slides

`App.jsx` manages:

- creating, inserting, duplicating, reordering, and deleting slides
- editing slide titles
- setting slide backgrounds
- attaching slide background audio
- selecting the active slide
- rendering block-specific editors with a switch by `block.type`

### Preview

The preview flow posts the current course JSON to `/api/preview-html`. The backend converts the authoring slides into runtime data using `build_course_definition()` and returns the full runtime HTML generated by `generate_runtime_html()`. The frontend then shows that HTML inside an iframe.

### Export

The editor supports:

- SCORM 1.2 export via `/api/export/scorm`
- SCORM 2004 export via `/api/export/scorm-2004`
- xAPI export on the backend side

Before export, the frontend runs lightweight validation such as missing video URLs, blank text blocks, and invalid fill-in-the-blank answers.

## 5. Import Features

### PowerPoint Import

- Frontend sends `.pptx` files to `/api/upload/pptx`
- Backend parses slides in `pptx_parser.py`
- Extracted output can include:
  - slide title
  - text blocks
  - lists
  - tables
  - images
  - slide background color/image
  - transitions
  - embedded audio
  - speaker notes

Important detail: PowerPoint colors are not always stored as direct RGB values. The parser reconstructs theme inheritance, slide/master color mapping, and fallback text styles before outputting HTML styles.

### PDF Import

- Frontend sends `.pdf` files to `/api/upload/pdf`
- Backend uses `pdf_parser.py`
- Typical extraction result: heading + body text + inline images where possible

### Storyline Import

- Backend exposes `/api/upload/story`
- `story_parser.py` extracts readable content blocks from `.story` files
- This import path exists on the backend even if not surfaced prominently in the current frontend

### Audio Upload

- Audio blocks upload files to `/api/upload/audio`
- Backend stores them in memory / media-serving flow and returns a `mediaId` plus preview URL
- Export later resolves those references into packaged media files

## 6. Shared Editing Features

### Rich Text Editor

`RichTextEditor.jsx` is used across multiple blocks. It supports:

- bold / italic / underline
- font family
- font size
- text color
- unordered lists
- alignment controls
- plain-text paste sanitization

This editor stores HTML, not plain text. That is why text-capable blocks can preserve inline formatting during preview and export.

### Mandatory / Optional Completion

Assessment and some media blocks expose a completion selector using `MandatorySelect.jsx`. This changes only the boolean completion flag on the block:

- `mandatory`
- or `isMandatory`

The selector affects runtime completion logic but does not alter the block’s core content.

### Marks

Assessment blocks typically include a numeric `marks` field. These marks are used when the runtime computes learner scoring and pass/fail evaluation.

## 7. Block-by-Block Reference

### Heading

- Type: `heading`
- Data: `content`, `headingLevel`
- Purpose: large semantic heading block
- Render path: becomes runtime `heading` with level 1-6
- Notes: editor uses rich text styling, runtime preserves heading hierarchy

### Text

- Type: `text`
- Data: `content` (HTML)
- Purpose: freeform rich text content
- Render path: passed through largely as authored HTML
- Notes: also used for AI-generated content and imported PPT/PDF text

### Image

- Type: `image`
- Data typically includes `image`, `imageUrl`, `src`, `caption`, `alt`, crop state
- Purpose: standalone visual block
- Features:
  - local upload
  - URL-based image support
  - crop workflow using `react-image-crop`
  - caption support
- Runtime: serialized to `image` component with `src`, `alt`, `caption`, and width

### Button

- Type: `button`
- Data: `content`, `targetSlideId`, `alignment`, `color`
- Purpose: navigational call-to-action block
- Features:
  - editable label
  - left / center / right alignment
  - target slide selection
  - custom button color
- Runtime: becomes a clickable button that can jump to another slide

### Flashcard

- Type: `flashcard`
- Data: `front`, `back`, optional `color`
- Purpose: flip-card learning interaction
- Features:
  - front/back authoring
  - color-driven visual theme
  - gradient and contrast generation
- Runtime: rendered as a two-sided card with flip behavior and derived theme colors

### Video

- Type: `video`
- Data: `videoUrl`, `isLocal`, `fileName`, `mandatory`
- Purpose: embed or upload a normal video asset
- Features:
  - YouTube URL support
  - Vimeo URL support
  - direct file upload
  - mandatory / optional completion state
- Runtime: decides between iframe embed and native `<video>` playback

### Interactive Video

- Type: `interactive-video`
- Data: `videoUrl`, `interactions`, `mandatory`
- Purpose: quiz checkpoints during video playback
- Features:
  - upload local video
  - create timestamped quiz interactions
  - per-interaction answer validation
  - optional “must answer correctly to continue” flow
- Runtime:
  - monitors playback time
  - pauses at trigger points
  - shows quiz overlay
  - resumes only after allowed completion

### Image Hotspot

- Type: `image-hotspot`
- Data: `imageUrl`, `hotspots[]`
- Purpose: annotated image with clickable points
- Hotspot data:
  - `id`
  - `x`, `y`
  - `title`
  - `content`
- Features:
  - click image to place hotspots
  - edit title/body per hotspot
- Runtime: displays hotspot markers and popup content

### Image Stack

- Type: `image-stack`
- Data: `slides[]`
- Purpose: card-like stack mixing images and micro-quizzes
- Slide item types:
  - `image`
  - `quiz`
- Features:
  - multiple image panels
  - optional captioning
  - quiz slide insertion between images
- Runtime: serializes to its own slide array and handles stepwise progression

### List

- Type: `list`
- Data: `items[]`
- Purpose: simple bullet list authoring
- Features: lightweight list item editing
- Runtime: becomes a list component

### Quote

- Type: `quote`
- Data: `content`, `author`, `layout`, `bgImage`, `bgOverlay`
- Purpose: styled quotation / pull quote
- Features:
  - multiple author placement layouts
  - optional background image
  - adjustable overlay darkness
- Runtime: exported as a styled quote scene

### Process

- Type: `process`
- Data: `steps[]` where each step has `title` and `content`
- Purpose: multi-step explanatory sequence
- Features:
  - step navigation in editor
  - add / delete / rename steps
  - rich text content per step
- Runtime: rendered as sequential process steps

### Audio

- Type: `audio`
- Data: `label`, `audioUrl`, `mediaId`, `isUploading`, `mandatory`
- Purpose: uploaded standalone audio asset
- Features:
  - upload MP3/WAV/OGG
  - authoring-time preview player
  - removable asset
- Runtime: packaged as media and rendered as an audio player block

### Table

- Type: `table`
- Data: `headers[]`, `rows[][]`, `tableColor`, `headerColor`
- Purpose: editable tabular content
- Features:
  - add/remove rows
  - add/remove columns
  - rich text cell editing through popup editor
  - separate body/header colors
- Runtime: converted to a display table with preserved cell content

### Columns

- Type: `columns`
- Data: `columns[][]`
- Purpose: multi-column layout container
- Child block support:
  - text sub-blocks
  - image sub-blocks
- Features:
  - add/remove columns
  - add/remove content inside each column
- Runtime: serialized into nested column components

### Tabs

- Type: `tabs`
- Data: `tabs[]` where each tab has `title`, `content`, `image`
- Purpose: tabbed reveal interface
- Features:
  - add/remove tabs
  - rich text content per tab
  - optional image per tab
- Runtime: exported as a tab-switching content module

### Accordion

- Type: `accordion` / `accordian`
- Data: `topics[]`
- Purpose: expandable grouped content sections
- Topic structure:
  - topic title
  - ordered content items
  - content item types: text or image
- Features:
  - reorderable items
  - per-topic mixed media
- Runtime: rendered as expandable/collapsible sections

### Scenario

- Type: `scenario`
- Data: `slides[]`
- Purpose: branching visual scenario with positioned dialogue hotspots
- Scenario slide structure:
  - `isErrorSlide`
  - `imageSrc`
  - `dialogues[]`
- Dialogue structure:
  - `text`
  - `x`, `y`
  - `action` = `next`, `error`, `restart`, or `static`
- Runtime: allows branched progression and error/restart paths

### Canvas

- Type: `canvas`
- Data: `items[]`, `canvasBg`
- Purpose: freeform arrangement of shapes and text
- Supported item types:
  - `rect`
  - `circle`
  - `triangle`
  - `text`
- Features:
  - drag/resize
  - z-index ordering
  - per-item style controls
  - shape and text authoring in one canvas
- Runtime: serializes positioned canvas items with geometry and styling

### Statement

- Type: `statement`
- Data: `image`, `textLayers[]`, `imageHeight`
- Purpose: image-backed statement/composite message block
- Features:
  - upload background image
  - place draggable rich-text overlays
  - resize overall image height
- Runtime: renders positioned text over image

### MCQ

- Type: `quiz`
- Data: `question`, `options[]`, `correctAnswer`, `marks`, `mandatory`
- Purpose: standard single-correct multiple choice
- Features:
  - add/remove options
  - select correct answer index
  - assign marks
- Runtime: scored single-answer quiz interaction

### True / False

- Type: `true_false`
- Data: `question`, `correctAnswer`, `isMandatory`, `marks`
- Purpose: binary assessment item
- Runtime: scored true/false interaction

### Fill in the Blank

- Type: `fill_blanks`
- Data: `question`, `answer`, `answers[]`, `caseSensitive`, `isMandatory`, `marks`
- Purpose: blank completion assessment
- Features:
  - automatic answer-slot normalization based on `____`
  - multiple blanks
  - case-sensitive toggle
- Runtime: validates each learner blank against normalized answers

### Multi-Select

- Type: `multi_select`
- Data: `question`, `options[]`, `correctAnswer[]`, `mandatory`, `marks`
- Purpose: multiple-correct selection question
- Runtime: validates a set of selected answers

### Matching

- Type: `matching`
- Data: `question`, `pairs[]`, `mandatory`, `marks`
- Purpose: pair left/right matching prompts
- Runtime: compares learner matches against authored pairs

## 8. Runtime Conversion Rules

`scorm_builder.py` is the bridge between editor blocks and runtime components. The `_block_to_component_raw()` function is the most important conversion layer. It:

- normalizes authoring block shapes
- renames fields where runtime needs a cleaner schema
- converts URLs to embeddable forms
- clamps coordinates and sizes for interactive layouts
- preserves mandatory flags and scoring metadata
- prepares nested structures such as columns, scenarios, and interactive videos

After conversion, the course is rendered as slides, layers, and components for the browser runtime.

## 9. Packaging and Media Handling

During export:

1. The frontend sends the full authoring course JSON.
2. The backend builds a normalized course definition.
3. `_extract_media_from_course()` walks the course definition.
4. Any data-URI media is written out as real files under packaged media paths.
5. `generate_manifest()` or `generate_manifest_scorm2004()` creates the LMS manifest.
6. `generate_runtime_html()` creates `index.html`.
7. Runtime JS and serialized course data are included as separate assets.

This split avoids LMS CSP issues and keeps packaged content self-contained.

## 10. AI and Media Generation Features

### AI Text Generation

- Frontend posts prompt + target block type to `/api/ai/generate`
- Backend uses Gemini to generate educational content
- Frontend inserts the result as a text block

### AI Image Generation

- Backend exposes `/generate-image`
- Uses an enterprise Gemini image endpoint
- Stores result in local `media/`
- Can be used as a source asset for visual blocks

### Unsplash Download Proxy

- Backend exposes `/unsplash-download`
- Fetches image bytes and returns a data URI
- Lets the editor bring third-party visuals into authoring format

## 11. Important Implementation Notes

- The editor stores rich text as HTML. Any block using `RichTextEditor` needs HTML-safe handling downstream.
- Some blocks use `mandatory`; others use `isMandatory`. This is legacy-compatible but worth standardizing in the future.
- Block IDs are regenerated during duplicate/clone flows to avoid collisions in nested structures.
- Imported PPTX/PDF slides are appended to the existing course rather than replacing it.
- Preview uses the same runtime builder as export, which is good for parity.
- The code already supports backend story import and xAPI export even if those paths are less visible in the main frontend UI.

## 12. Suggested Future Cleanup Areas

- Standardize block completion fields to one name.
- Standardize image source field names across blocks (`image`, `imageUrl`, `src`, etc.).
- Break `App.jsx` into smaller orchestration hooks or modules.
- Add a formal schema for authoring blocks to reduce drift between frontend and backend.
- Add automated preview/export regression tests for each block type.
